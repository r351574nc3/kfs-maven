32  /**
33   * This interface defines the methods that a LockService must provide. LockServiceImpl consists of methods that manage the various
34   * locks used in the Budget module. Locks are needed to serialize user updates since a BC Edoc is potentially editable by many users
35   * simultaneously and the default Optimistic locking scheme used by KFS would produce an inconsistent set of data. <B>Accountlock</B>
36   * controls exclusive access to the BC Edoc <B>Positionlock</B> controls exclusive access to a BC Position <B>Fundinglock</B>
37   * controls shared funding access. An associated Positionlock must exist before attempting to get a Fundinglock. Accountlock and
38   * Fundinglock are mutex. <B>Transactionlock</B> controls exclusive access to serialize updates to the accounting lines in the BC
39   * Edoc. A Fundinglock must exist before creating a Transactionlock. The Transactionlock lifecycle is short, required only for the
40   * duration of the accounting line update.
41   */
42  public interface LockService {

45       * This method attempts to lock the given Account for the passed in uuid. Finding an exising account lock for the uuid returns
46       * success without having to relock After setting an accountlock, if any funding locks are found, it releases the accountlock
47       * and sets BCLockStatus with LockStatus.FLOCK_FOUND. Accountlocks and Fundinglocks are mutex
48       * 
49       * @param bcHeader
50       * @param principalId
51       * @return BudgetConstructionLockStatus with lockStatus.SUCCESS, OPTIMISTIC_EX (lost optimistic lock), FLOCK_FOUND (also sets
52       *         fundingLocks), BY_OTHER (also sets accountLockOwner), NO_DOOR (null bcHeader)
53       */
54      public BudgetConstructionLockStatus lockAccount(BudgetConstructionHeader bcHeader, String principalId);
